# 스마트화 증진 로드맵 (규칙·집계 기반)

- 작성일: 2026-07-01
- 유형: 로드맵 / 설계 스펙 (여러 아이디어 우선순위)
- 범위: **AI 비용 없음.** 이미 쌓인 데이터를 규칙·집계·파생계산으로 "스마트하게" 보여주고 자동화한다.
- 우선가치(사용자 선택): ① 학생 이탈·성적하락 조기감지·개입 · ② 관리자 반복업무 자동화 · ③ 학생 동기·참여

## 1. 목표와 우선순위 기준

`ROI = 임팩트 ÷ 구현비용`. 각 아이디어에 우선가치 태그(①/②/③)를 붙이고, **재활용도**(기존 모듈/데이터 재사용)가 높을수록 앞선다.

핵심 통찰 — **"학생 건강지수(Health Score)" 엔진이 세 가치의 공통 토대다.** 여러 신호를 하나의 위험지수로 합치면 ①위험 감지, ②일일 브리핑 재료, ③약점 기반 미션/넛지까지 전부 이 엔진 하나를 재사용한다. 그래서 Wave 1의 주춧돌로 둔다.

### 규칙기반을 택한 이점 (마이그레이션 부담 최소)
대부분 **파생계산**이라 신규 컬럼/테이블이 거의 필요 없다.
- 가중치·임계값 → 기존 `app_settings` 패턴에 저장 (원장이 조정 가능, 마이그레이션 불필요)
- 스트릭 최고기록·뱃지 지급 로그 → 기존 `specialNote` JSON envelope(`SpecialNoteEnvelope`)에 키 추가 (컬럼 추가 불필요)
- 자동 알림 → 기존 관리자 인박스 / 학생 메시지 스레드 재사용
- 디제스트/트리거 중복방지 → 날짜 기준 멱등 키

> 관련 레포 함정 감사: 저장 정합성은 [integrity-check], 미션/쿠폰 멱등성은 [mission-check], UI는 [glass-check], 새 컬럼이 실제로 생기면 [migration-check]로 반드시 교차검증.

## 4. Wave 1 상세 설계

### 4.1 학생 건강지수 엔진 (#1) — 주춧돌
- 위치: `lib/health-score.ts` — **순수 함수 모듈** (`absence-stats.ts` 패턴, TDD 대상).
- 입력: 위 §2 신호들 (각 학생, 최근 N일 윈도우).
- 출력:
  - `score` 0~100 (높을수록 위험)
  - `band`: 정상 / 주의 / 위험
  - `factors[]`: 어떤 신호가 얼마나 기여했는지 breakdown (관리자가 "왜 위험한지" 즉시 파악)
- 가중치·임계값: `app_settings`에 저장해 원장이 조정 (기본값은 스펙에 제안치 명시). 마이그레이션 불필요.
- 재사용처: #2 디제스트, #3 트리거, #8 미션 추천이 이 출력을 그대로 소비.

### 4.2 일일 자동 브리핑 (#2)
- cron: 기존 sweep 크론 옆에 `daily-digest` 잡 추가 (동일 인증 패턴).
- 산출물: 관리자 인박스 카드 1건/일 — 어제 결석·이탈급증 명단, 위험밴드 신규 진입, 상담 노쇼, 승인대기 건수, 스트릭 끊긴 위험학생.
- **멱등성**: 날짜 기준 1회만 생성(중복 방지 키). [mission-check]식 멱등성 원칙 준수.

### 4.3 연속 결석·이탈 트리거 (#3)
- `absence-stats` 확장: 연속 결석 임계(기본 2일 연속) 및 이탈 급증 감지.
- 액션: 관리자 인박스 알림 자동 + (설정 시) 학생 메시지 스레드 자동 발송.
- 멱등성: 동일 학생·기간 중복 알림 방지.

### 4.4 학생 미션 페이지 = "오늘 할 일 허브" (#4)
- 라우트: `/student/missions`. 홈에서 "미션" 진입 강조 (공부한 걸 홈에서 찾아 헤매지 않고 여기서 할 일을 딱 봄).
- 구성(전부 기존 데이터 재사용):
  1. **상단 스트릭** — 듀오링고식 연속출석 (§4.5)
  2. **오늘 계획(진도)** 항목 체크
  3. **체크리스트** — 휴대폰 제출·수면 기록
  4. **설정형 쿠폰 미션** 진행 (Wave 2 #8에서 개인화 추천 확장)
- 완료 시 즉시 피드백 애니메이션.
- 디자인: iOS26 Liquid Glass 규칙 준수 — `.glass` 유틸 재사용, 색=의미/역할, **보라·인디고 금지**, 한글 `word-break` 세로깨짐 주의. ([glass-check])

### 4.5 듀오링고식 연속출석 스트릭 (#4의 얼굴)
- 계산: `lib/streak.ts` — **순수 함수 모듈** (TDD 대상).
  - 등원일 집합에서 **일요일 제외** (센터 미운영일).
  - 연속 등원일 = 오늘부터 역순, 일요일은 skip(끊지 않음)하며 이어진 등원일 수.
  - **승인휴가/정당사유는 스트릭 유지**(끊지 않음) — 이탈순위에서 정당사유 제외하는 것과 일관.
  - 결석(등원 없음, 정당사유 아님)이면 리셋.
- UI: 불꽃 애니메이션(획득/유지/마일스톤). Glass 규칙 준수, 성취색은 따뜻한 계열이되 보라/인디고 금지.
- 저장: 현재 스트릭은 파생계산. **최고기록·마일스톤 지급 로그만** `specialNote` envelope에 키 추가(컬럼 불필요).
- 마일스톤(예: 7·14·30·100일)에서 템플릿 축하 넛지 자동 발송(AI 아님, 기존 messages 재사용).

## 5. 컴포넌트 경계 (독립·테스트 가능)

| 유닛 | 역할 | 의존 |
|---|---|---|
| `lib/health-score.ts` | 신호→위험지수(순수) | 신호 리더들, app_settings 가중치 |
| `lib/streak.ts` | 등원일→연속기록(순수, 일요일 제외) | 등원일 집합 |
| `absence-stats.ts` 확장 | 연속결석/이탈급증 판정 | 기존 |
| cron `daily-digest` | 브리핑 생성(멱등) | health-score, inbox |
| `/student/missions` 화면 | 할 일 허브 렌더/상호작용 | plan·checklist·미션엔진·streak (읽기) |

각 순수 모듈은 "무엇을 하나 / 어떻게 쓰나 / 무엇에 의존하나"가 명확하고, 내부를 바꿔도 소비자가 안 깨지도록 인터페이스로만 결합한다.
